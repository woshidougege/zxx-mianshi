🧪 测试基础知识

测试分类

1. 功能测试

定义：验证软件功能是否符合需求规格说明

测试方法：

  • 等价类划分：将输入划分为有效类和无效类
  • 边界值分析：测试边界附近的值（最大、最小、刚好超出）
  • 场景法：按照用户使用流程设计测试场景
  • 错误推测法：根据经验推测可能出错的地方

示例：登录功能测试

  • 正确的用户名和密码 → 登录成功
  • 错误的密码 → 提示密码错误
  • 不存在的用户 → 提示用户不存在
  • 密码输入3次错误 → 账号锁定
  • 特殊字符、SQL注入 → 安全测试

2. 接口测试

定义：测试系统之间的数据交互接口

测试要点：

  • 请求验证：URL、Method（GET/POST）、Headers、Parameters、Body
  • 响应验证：Status Code（200/404/500）、Response Body、Response Time
  • 业务逻辑：数据库数据是否正确、关联接口影响
  • 异常场景：参数缺失、参数类型错误、超时、权限校验

常用工具：Postman、JMeter、Python requests

3. 性能测试

定义：验证系统在特定负载下的响应时间和稳定性

核心指标：

指标 说明 目标值（参考）
QPS 每秒查询率 根据业务，一般 > 1000
RT 响应时间 接口 < 500ms，页面 < 3s
错误率 失败请求占比 < 0.1%
并发数 同时访问用户数 根据业务设定

测试类型：

  • 负载测试：逐步增加负载，找到系统瓶颈
  • 压力测试：超过系统能力，测试极限和恢复能力
  • 稳定性测试：长时间运行（如24小时），验证稳定性和内存泄漏

4. 自动化测试

定义：使用脚本代替手工执行重复性测试

适合自动化的场景：

  • 需求稳定，不频繁变动
  • 重复执行次数多（如回归测试）
  • 项目周期长，值得投入
  • 接口测试（稳定性高）

不适合自动化：

  • 需求频繁变化
  • 一次性测试
  • UI频繁变动的页面
  • 探索性测试

ROI计算：

  • 成本：开发时间 + 维护时间
  • 收益：节省的人工执行时间
  • ROI = 收益 / 成本

测试流程

完整测试流程

  1. 需求评审
    • 参与需求评审会议
    • 理解业务需求和技术方案
    • 识别风险点和测试难点
    • 提出测试建议
  2. 测试计划
    • 测试范围：测什么、不测什么
    • 测试策略：如何测（功能、接口、性能）
    • 测试时间：排期安排
    • 测试资源：人员、环境、工具
  3. 用例设计
    • 根据需求设计测试用例
    • 用例评审（与开发、产品确认）
    • 用例优化和补充
  4. 测试执行
    • 冒烟测试（基本功能验证）
    • 功能测试（详细功能验证）
    • 回归测试（验证修复不影响其他功能）
    • Bug管理（提Bug、跟踪、验证）
  5. 测试报告
    • 测试覆盖情况
    • Bug统计（总数、严重级别、遗留）
    • 测试结论（是否可以上线）
    • 风险提示
  6. 上线验收
    • 上线checkList
    • 线上验证（冒烟测试）
    • 线上监控
    • 问题应急响应

测试用例设计方法

等价类划分法

原理：将输入划分为若干等价类，每类选一个代表性测试

示例：年龄输入框（要求1-120）

等价类 测试数据 期望结果
有效等价类 50 输入成功
无效等价类（小于最小值） 0 提示错误
无效等价类（大于最大值） 121 提示错误
无效等价类（非数字） abc 提示错误

边界值分析法

原理：边界附近最容易出错，测试边界值和边界附近的值

示例：年龄输入框（要求1-120）

  • 最小值-1：0（无效）
  • 最小值：1（有效）
  • 最小值+1：2（有效）
  • 中间值：60（有效）
  • 最大值-1：119（有效）
  • 最大值：120（有效）
  • 最大值+1：121（无效）

场景法

原理：模拟用户真实使用场景

示例：电商下单场景

  1. 用户登录
  2. 浏览商品列表
  3. 查看商品详情
  4. 加入购物车
  5. 修改数量
  6. 选择地址
  7. 选择支付方式
  8. 提交订单
  9. 支付
  10. 查看订单

异常场景：

  • 库存不足时下单
  • 支付失败
  • 网络超时
  • 重复提交订单

Bug管理

Bug生命周期

New → Open → Fixed → Reopen/Verified → Closed

  • New: 新发现的Bug
  • Open: 确认Bug，开发处理中
  • Fixed: 开发已修复
  • Verified: 测试验证通过
  • Reopen: 验证未通过，重新打开
  • Closed: 关闭Bug

Bug严重级别

级别 说明 示例
P0/Blocker 致命，系统崩溃或核心功能不可用 系统崩溃、数据丢失、无法登录
P1/Critical 严重，主要功能受影响 支付失败、订单无法提交
P2/Major 一般，功能部分不可用 某个按钮不生效、提示信息错误
P3/Minor 轻微，体验问题 UI美观问题、文案错误

Bug描述规范

标题：[模块]简短描述问题

内容包含：

  1. 环境：测试/生产，浏览器版本，APP版本
  2. 前置条件：账号状态、数据准备
  3. 复现步骤：详细操作步骤（1、2、3...）
  4. 实际结果：出现了什么问题
  5. 期望结果：应该是什么样的
  6. 截图/日志：错误截图或日志

测试工具对比

工具 类型 优点 缺点 适用场景
Postman 接口测试 简单易用、图形化 批量测试功能弱 接口调试、快速验证
JMeter 性能/接口测试 功能强大、免费 学习成本高 性能测试、批量接口测试
pytest 自动化框架 简洁、功能强 需要编程基础 接口自动化、单元测试
Selenium UI自动化 跨浏览器 维护成本高 Web UI自动化
Fiddler 抓包工具 功能强大 仅Windows 抓包分析、Mock数据
Charles 抓包工具 跨平台 付费 移动端抓包、弱网测试
